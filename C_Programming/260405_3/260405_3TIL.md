1. 프로젝트 목표
요구사항: 메인/예비 서버의 헬스 점수를 입력받아, 과락(40점 미만) 여부와 총합 점수에 따라 가용성 상태(점검/안정/경고)를 판별하는 모니터링 로직 구현.
[실전 코딩 문제: AWS 이중화 서버(Multi-AZ) 헬스 체크 시스템]
요구사항: 메인 서버(AZ-A)와 예비 서버(AZ-B)의 상태 점수(Health Score)를 입력받아, 전체 시스템의 가용성을 평가하는 모니터링 로직을 작성하라.
입력: 두 서버의 상태 점수(정수, 0~100)를 공백으로 구분하여 입력받는다.
연산: 두 점수의 합인 총합 가용성 점수를 먼저 출력한다.
조건 판별 및 출력:
[장애 격리] 두 서버 중 단 하나라도 점수가 40점 미만인 서버가 있다면, 총합과 무조건 관계없이 "알림: 긴급 셧다운 및 점검"을 출력한다.
[안정] 위 조건에 해당하지 않고(즉, 둘 다 40점 이상), 총합 점수가 150점 이상이면 "상태: 안정(Stable)"을 출력한다.
[경고] 위 두 조건에 모두 해당하지 않으면 "상태: 로드밸런싱 필요(Warning)"을 출력한다.

초기에 짠 코드
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

int main()
{
	int AZ_A, AZ_B;
	int Health_Score;

	printf("서버 상태 점수 입력 (A B): ");
	scanf("%d %d", &AZ_A, &AZ_B);

	Health_Score = AZ_A + AZ_B;
	printf("총합 가용성 점수 : %d\n", Health_Score);

	if(AZ_A < 40 || AZ_B < 40)
	{
		printf("알림: 긴급 셧다운 및 점검");
	}
	else if (Health_Score >= 150)
	{
		printf("상태: 안정(Stable)");
	}
	else
	{
		printf("상태: 로드밸런싱 필요(Warning)");
	}

	return 0;
}
2. 초기 코드의 치명적 결함 (The Trap)
화살표 반패턴 (Arrow Anti-Pattern): 초기 로직(교수님 예제 참조)은 else 블록 안에 또다시 if-else가 깊게 파고드는 중첩(Nesting) 구조를 띠고 있었다. 이는 코드의 깊이를 깊게 만들어 가독성과 유지보수성을 크게 떨어뜨린다.
네이밍 컨벤션(Naming Convention) 위반: 변수명을 AZ_A, Health_Score 등 대문자로 작성했다. C/Java 등 실무 환경에서 대문자는 보통 변하지 않는 상수(Constant)나 클래스(Class)를 의미하므로, 타 개발자가 읽을 때 심각한 혼란을 초래할 수 있다.


3. 단계별 디버깅 과정
조건문 평탄화 (Flattening): 논리 연산자(||)를 활용하여 가장 치명적인 예외(40점 미만)를 먼저 걸러내고, 나머지 조건들을 if - else if - else의 1차원 폭포수 구조로 재배치하여 중첩을 완벽히 제거함.
변수명 표준화: 지역 변수(Local Variable)의 목적에 맞게 스네이크 케이스(az_a, health_score)를 적용하여 대소문자 규칙을 바로잡음.

4. 최종 최적화 코드 (Final Verified Code)
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

int main()
{
	int az_a, az_b; //AZ_A나 AZ_B를 az_a az_b로 수정 대문자는 암묵적으로 상수를 칭할때만 사용하기 때문이다.
	int health_score; //위에와 동일.

	printf("서버 상태 점수 입력 (A B): ");

	if(az_a < 40 || az_b < 40)
	{
		printf("알림: 긴급 셧다운 및 점검");
	}
	else if (health_score >= 150)
	{
		printf("상태: 안정(Stable)");
	}
	else
	{
		printf("상태: 로드밸런싱 필요(Warning)");
	}

5. 핵심 인사이트 (Takeaway)
가독성을 위한 조건문 평탄화: 코드는 오른쪽으로 깊어질수록 읽기 힘들어진다. 예외 상황을 먼저 if로 쳐내고 빠져나오는 방식으로 중첩을 최소화해야 한다.
네이밍은 협업의 기본: 변수명의 대소문자(Camel, Snake 등)는 단순한 취향 차이가 아니라, 데이터의 성격(변수냐 상수냐)을 규정하는 엄격한 약속(Convention)임을 명심할 것.
