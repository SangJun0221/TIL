# 260410_3 C언어 트러블슈팅: Sentinel 값 제어와 조건부 누적 파이프라인

## 1. 문제 요약
정수를 무한으로 입력받되, 특정 종료 신호(Sentinel Value)가 들어오면 입력을 차단하고, 유효한 데이터 중 짝수만 선별하여 합산하는 파이프라인 구축.

**[처리 조건]**
* 입력값이 `-1`이면 즉시 루프를 종료한다. (합계 제외)
* 입력된 유효 데이터 중 **짝수**만 합계에 누적한다.
* 입력 종료 후 최종 짝수의 합을 출력한다.

---

## 2. 초기 구현 및 최종 코드 (Verified Code)
초기부터 Sentinel 패턴과 데이터 필터링의 순서를 완벽하게 분리하여, 별도의 에러 없이 요구사항을 100% 충족했다.

```c
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

int main()
{
	int a;
	int sum = 0; // 누적 변수 초기화

	for (;;) {
		scanf("%d", &a);

		// [Phase 1: Sentinel 검사 (Guard Clause)]
		// 데이터가 파이프라인에 진입하기 전, 종료 신호인지 먼저 확인하여 차단
		if (a == -1) {
			break;
		}

		// [Phase 2: 데이터 필터링 및 누적 처리]
		// 유효성이 검증된 데이터에 한해서만 짝수 여부 판별
		if (a % 2 == 0) {
			sum = sum + a;
		}
	}

	printf("%d\n", sum);

	return 0;
}
```

## 3. 아키텍처 분석 (Why it works)
이 코드가 결함 없이 견고하게 동작하는 이유는 명확한 파이프라인 순서 제어에 있다.

데이터 오염 방지: 만약 a % 2 == 0 (짝수 판별)이나 누적 로직을 먼저 배치했다면, 시스템 제어를 위한 신호인 -1이 일반 데이터처럼 취급되어 로직에 혼선을 줄 위험이 있었다.

관문(Guard)의 올바른 배치: 입력을 받은 직후 가장 먼저 a == -1을 검사하여 무의미한 연산을 사전에 차단(Break)했다. 이후 종료값이 아닐 때만 다음 분기로 내려가 비즈니스 로직(짝수 판별 및 누적)을 수행하도록 격리했다.

## 4. 핵심 인사이트 (Takeaway)
Sentinel Value의 절대 원칙: 입력의 끝을 알리는 Sentinel 값은 반드시 입력 직후 최우선으로 검사해야 한다. 제어 신호와 일반 데이터가 로직 안에서 섞이는 순간 치명적인 버그가 발생한다.

"무엇을 제외할 것인가"가 먼저다: 누적합 문제를 풀 때는 조건에 맞는 데이터를 찾는 것보다, 파이프라인에 들어오면 안 되는 예외 데이터(음수, -1, 문자 등)를 최상단에서 먼저 걸러내는(Filtering) 설계가 필수적이다.
