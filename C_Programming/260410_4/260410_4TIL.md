# 260410_4 C언어 트러블슈팅: Max/Min 동적 초기화 및 불필요한 상태 변수(Phantom Variable) 제거

## 1. 문제 요약
정수를 무한으로 입력받아, 입력된 유효 데이터 중 최댓값(max)과 최솟값(min)의 차이를 계산하는 로직 구현.

**[처리 조건]**
* 입력값이 `-1`이면 즉시 입력을 종료한다. (`-1`은 계산에서 제외)
* 처음 입력된 '정상 값'으로 `max`, `min`을 초기화한다.
* 이후 입력값과 비교하여 최댓값, 최솟값을 동적으로 갱신한다.
* 출력: `최댓값 - 최솟값` 결과 출력

---

## 2. 초기 구현 (The Trap)
```c
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

int main()
{
	int a;
	int max, min;
	int first = 1;
	int sum = 0; // [문제의 시작]

	for (;;) {
		scanf("%d", &a);

		if (a == -1) break;

		if (first == 1) {
			max = a; min = a;
			first = 0;
		}
		else {
			if (max < a) max = a;
			if (min > a) min = a;
		}
	}

	sum = sum + a; // [무의미한 연산]

	printf("차이 = %d\n", max - min);

	return 0;
}
```

## 3. 문제점 분석 (Logical Flaw)
목적 잃은 변수(Phantom Variable)의 잔재: 이전 문제(누적합)를 풀던 관성 때문에, 이 문제와 전혀 무관한 sum 변수를 선언하고 연산(sum = sum + a)까지 시도했다. 반복문이라고 해서 무조건 합계를 구하는 것이 아님을 명심해야 한다.

불필요한 메모리와 연산 낭비: 반복문이 끝난 시점의 a는 무조건 제어 신호인 -1이다. 이를 sum에 더하는 행위는 논리적으로 완벽한 오류이며, 실무에서는 메모리와 사이클을 낭비하는 데드 코드(Dead Code)로 분류된다.

## 4. 엣지 케이스 발견 (Edge Case Discovery)
문제 제기: 만약 유저가 프로그램을 실행하자마자 첫 입력으로 바로 -1을 넣는다면?

결과: 반복문이 즉시 종료되고, max와 min은 한 번도 초기화되지 않은 상태(Garbage Value)에서 max - min 연산이 실행되어 치명적인 런타임 에러(Runtime Error)를 뿜어낸다.

해결책: 반복문 종료 후 출력 단계에서, 상태 변수인 first가 여전히 1인지(즉, 유효 데이터가 한 번이라도 들어왔는지) 검사하는 방어 로직이 필요하다.

## 5. 최종 최적화 코드 (Final Verified Code)
불필요한 sum 변수를 완전히 제거하고, 데이터가 아예 입력되지 않은 엣지 케이스까지 완벽하게 방어하는 강건한(Robust) 아키텍처로 재작성했다
```c
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

int main()
{
	int a;
	int max, min;
	int first = 1;

	for (;;) {
		scanf("%d", &a);

		// 1. Guard Clause (종료 신호 차단)
		if (a == -1) break;

		// 2. 초기화 파이프라인
		if (first == 1) {
			max = a;
			min = a;
			first = 0; // 초기화 완료 상태로 변경
		}
		// 3. 갱신 파이프라인
		else {
			if (max < a) max = a;
			if (min > a) min = a;
		}
	}

	// 4. 안전한 출력 (Edge Case 방어 로직)
	if (first == 1) {
		// 유효한 데이터가 단 하나도 들어오지 않은 경우
		printf("Error: 입력된 정상 데이터가 없습니다.\n");
	} 
	else {
		// 정상적으로 데이터가 1개 이상 들어온 경우
		printf("차이 = %d\n", max - min);
	}

	return 0;
}
```

## 6. 핵심 인사이트 (Takeaway)
관성 탈피와 목적 지향 설계: 반복문 문제를 조우했다고 관성적으로 sum을 선언하지 마라. 현재 비즈니스 로직에서 실제로 추적하고 갱신해야 하는 상태(State)가 무엇인지 정확히 타겟팅해야 한다.

동적 초기화 패턴: 배열을 사용하지 않는 Max/Min 찾기 로직에서는 임의의 숫자(0 등)로 초기화하면 음수 입력 시 버그가 발생한다. 반드시 첫 번째 유효 입력값으로 초기화(first 플래그 활용)하는 패턴을 체화해야 한다.

엣지 케이스(Edge Case) 방어: 코드는 '잘 될 때'가 아니라 '안 될 때'를 대비해서 짜야 한다. 데이터가 단 하나도 들어오지 않고 종료되는 최악의 시나리오까지 고려한 코드가 진짜 엔지니어의 코드다.
