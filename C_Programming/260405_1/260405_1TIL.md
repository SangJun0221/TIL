프로젝트 목표
[실전 코딩 문제: AWS EC2 서버 CPU 사용량 모니터링]
요구사항: 현재 가동 중인 3대의 EC2 인스턴스 CPU 사용량(%)을 입력받아, 평균 부하를 계산하고 오토 스케일링(Scale-Out / Scale-In) 여부를 판별하는 프로그램을 작성하라.
입력: 3대의 서버 CPU 사용량(정수형, 0~100)을 공백으로 구분하여 입력받는다.
연산 및 출력 1: 3대의 평균 CPU 사용량을 소수점 둘째 자리까지 출력한다.
조건 판별 및 출력 2:
평균이 75.00 이상이면 "Warning: Scale-Out Required" (서버 증설 필요)
평균이 30.00 미만이면 "Warning: Scale-In Required" (서버 감축 필요)
그 외의 범위라면 "Status: Normal" (정상)을 출력한다.

초기에 짠 코드
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

int main()
{
int a1, a2, a3;
float b;

printf("CPU 사용량 입력 (3대): ");
scanf("%d %d %d", &a1, &a2, &a3);

b = (a1 + a2 + a3) / 3.0f;
printf("평균 : %.2f\n", b);

if (b >= 75)
{
printf("Warning: Scale-Out Required");
}
else if (b < 30)
{
printf("Warning: Scale-In Required");
}
else
{
printf("Status: Normal");
}

return 0;
}

2. 초기 코드 및 한계점
현재 코드는 정상 작동하지만, 2031년 Java 백엔드 엔지니어 관점에서 볼 때 유지보수성과 타입 안정성 측면에서 두 가지 결함이 존재한다.
암시적 형변환(Implicit Casting) 의존: if (b >= 75)에서 변수 b는 실수형(float)이나, 비교 대상인 75는 정수형(int)이다. C언어 컴파일러가 이를 실수로 자동 변환해주지만, Java와 같이 엄격한 정적 타입(Static Type) 언어에서는 이러한 타입 불일치가 예상치 못한 버그를 유발할 수 있다.
변수명 네이밍 룰(Naming Rule) 위반: a1, a2, a3, b와 같은 의미 없는 알파벳 나열은 타인이 코드를 읽을 때 가독성을 심각하게 저해한다.

3. 단계별 디버깅 과정
타입 명시화 (Explicit Casting): 조건문의 정수 상수 75와 30을 실수형인 75.0f, 30.0f로 명확하게 변경하여 좌항과 우항의 데이터 타입을 일치시킴.
직관적인 변수명 적용: 입력 변수는 cpu1, cpu2, cpu3로, 결과 변수는 avg_cpu로 변경하여 데이터의 목적을 명확히 함.

4. 최종 최적화 코드 (Final Verified Code)
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

int main()
{
int cpu1, cpu2, cpu3; // 단순 알파벳이 아닌 직관적인 변수로 변경
float avg_cpu; //위에 a1 a2 a3와 동일 문제.

printf("CPU 사용량 입력 (3대): ");
scanf("%d %d %d", &cpu1, &cpu2, &cpu3);

avg_cpu = (cpu1 + cpu2 + cpu3) / 3.0f;
printf("평균 : %.2f\n", avg_cpu);

if (avg_cpu >= 75.0f) //똑똑한 컴파일러 경우 자동으로 실수 75.0으로 변환해주겠지만 주력으로 다룰려는 Java는 엄격하게 표시해야하기에 그에 대해 수정.
{
printf("Warning: Scale-Out Required");
}
else if (avg_cpu < 30.0f) //앞선 75와 동일 문제.
{
printf("Warning: Scale-In Required");
}
else
{
printf("Status: Normal");
}

return 0;
}

5. 핵심 인사이트 (Takeaway)
명시적 데이터 타입 일치: 컴파일러의 자동 변환에 기대지 않고, 데이터 타입을 의도적으로 일치시키는 습관(예: 75 -> 75.0f)은 런타임 에러를 방지하는 방어적 프로그래밍의 기본이다.
가독성은 곧 성능이다: 코드는 컴퓨터가 실행하지만, 유지보수는 사람이 한다. 변수명만 봐도 로직이 이해되도록 직관적인 네이밍을 생활화해야 한다.
