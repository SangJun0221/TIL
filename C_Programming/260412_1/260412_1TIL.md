# 260412_1 C언어 트러블슈팅: 숫자 야구 게임의 상호 배타적 조건 분기(Mutual Exclusivity)

## 1. 문제 요약
컴퓨터가 생성한 3자리의 난수와 사용자의 입력값을 비교하여 Strike, Ball, Out을 판정하는 로직 구현.

**[처리 조건]**
* 난수: 1~9 사이의 서로 다른 숫자 3개 생성 (중복 허용 안 함)
* Strike: 자리와 숫자가 모두 일치
* Ball: 숫자는 일치하나 자리가 다름
* Out: 일치하는 숫자가 전혀 없음
* `3S` 달성 시 무한 루프 종료

---

## 2. 아키텍처 분석 (Logical Flaw vs. Best Practice)

숫자 야구 로직을 짤 때 초보자들이 가장 많이 저지르는 두 가지 치명적 결함을 완벽하게 방어한 설계다.

**① 조건의 상호 배타성 (Mutual Exclusivity) 확보**
* **위험:** 하나의 입력값(`c1`)에 대해 `if(Strike 조건)`과 `if(Ball 조건)`을 독립적으로 걸면, 하나의 숫자가 Strike이면서 동시에 Ball로 중복 카운팅되는 심각한 논리 오류가 발생한다.
* **해결:** `if - else if - else` 구조를 사용하여 파이프라인을 **상호 배타적**으로 묶었다. Strike 관문을 통과하면 이후 조건은 무시되며, 둘 다 아니면 무조건 Out으로 빠지게 하여 데이터 오염을 원천 차단했다.

**② 트랜잭션별 상태 초기화 (State Reset per Transaction)**
* **위험:** 루프 바깥에서 `s, b, o`를 한 번만 0으로 초기화하면, 매 입력마다 이전 판정 결과가 계속 누적되어 10S 15B 같은 쓰레기값(Garbage Data)이 출력된다.
* **해결:** `while(1)` 루프 진입 직후, 새로운 입력(`scanf`)을 받기 전에 카운터 변수(`s=0; b=0; o=0;`)를 매번 초기화하여 각 턴(Turn)의 독립성을 보장했다.

---

## 3. 최종 최적화 코드 (Final Verified Code)

출력 가독성을 위해 개행 문자(`\n`)를 추가하여 UI 측면을 보완했다.

```c
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>
#include <stdlib.h>
#include <time.h>

int main()
{
	int a1, a2, a3; // 컴퓨터의 난수
	int c1, c2, c3; // 사용자의 입력값
	int s, b, o;    // 판정 카운터

	srand(time(NULL));

	// 1. 중복 없는 난수 생성 파이프라인
	for (;;) {
		a1 = rand() % 9 + 1;
		a2 = rand() % 9 + 1;
		a3 = rand() % 9 + 1;

		if (a1 != a2 && a1 != a3 && a2 != a3) {
			break;
		}
	}

	// 2. 판정 파이프라인
	while (1) {
		// [핵심] 매 턴마다 상태(State) 초기화
		s = 0; b = 0; o = 0;

		scanf("%d %d %d", &c1, &c2, &c3);

		// [c1 검사] 상호 배타적 분기
		if (c1 == a1) { s++; }
		else if (c1 == a2 || c1 == a3) { b++; }
		else { o++; }

		// [c2 검사] 상호 배타적 분기
		if (c2 == a2) { s++; }
		else if (c2 == a1 || c2 == a3) { b++; }
		else { o++; }

		// [c3 검사] 상호 배타적 분기
		if (c3 == a3) { s++; }
		else if (c3 == a1 || c3 == a2) { b++; }
		else { o++; }

		// 가독성을 위한 개행(\n) 추가
		printf("%dS %dB %dO\n\n", s, b, o);

		// 종료 조건 (Guard Clause)
		if (s == 3) {
			break;
		}
	}

	return 0;
}
```

## 4. 핵심 인사이트 (Takeaway) & Action Item
분기문의 특성 이해: 여러 개의 조건을 다룰 때, 독립적으로 판정해야 하는지(독립 if) 아니면 하나만 걸러내야 하는지(if - else if) 데이터의 생명주기를 먼저 파악해야 한다.

트랜잭션 고립성: 반복문 내에서 값이 누적되는 변수는 반드시 **"어느 시점에 초기화(Reset)되어야 하는가"**를 손계산(Dry Run)으로 검증하는 습관을 들여야 한다. 실무 백엔드에서 사용자 요청(Request) 간에 데이터가 섞이는 것을 막는 기본 원리다.
