# 260410_1 C언어 트러블슈팅: 출금 후 수수료 적용 조건 처리

## 1. 문제 요약
기존 잔액과 출금 금액을 입력받아, 잔액 부족 여부를 검증하고 조건부 수수료를 차감하는 파이프라인 구축.

**[처리 조건]**
* 출금 금액이 잔액보다 크면 `잔액 부족` 출력
* 출금이 가능하면 출금 후 남은 잔액 계산
* 출금 후 남은 잔액이 1000원 미만이면 수수료 500원 추가 차감
* 수수료까지 차감한 결과가 0원 이상일 때만 출금 성공, 그렇지 않으면 `잔액 부족` 출력

---

## 2. 초기 구현 (The Trap)
```c
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

int main()
{
	int money, change;
	int balance;

	scanf("%d %d", &money, &change);

	balance = money - change;

	if (balance < 1000) {
		balance = balance - 500;
	}

	if (balance < 0) {
		printf("잔액 부족");
		return 0;
	}
	else {
		printf("출금 성공, ");
	}

	printf("잔액 : %d", balance);

	return 0;
}
```

## 3. 문제점 분석 (Logical Flaw)
문제점 1: 출금 자체가 불가능한 경우(예: 잔액 0원, 출금 10000원)에도 무작정 빼기 연산을 한 뒤, 수수료 차감 조건(balance < 1000)을 검사하도록 작성되었다. 마이너스 통장에 수수료를 매기는 꼴이다.

문제점 2: 결과는 운 좋게 맞게 나오지만, 비즈니스 로직의 순서가 거칠다. '출금 가능 여부 1차 판정 ➔ 수수료 적용 ➔ 최종 잔액 2차 검사' 순서로 파이프라인을 타는 것이 훨씬 강건한 아키텍처다.

# 4. 최종 최적화 코드 (Final Verified Code)
가드 절(Guard Clause)을 도입하여, 안 되는 조건은 즉시 쳐내고 통과한 데이터만 수수료 파이프라인에 진입하도록 재설계했다.
```c
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

int main()
{
	int money, change;
	int balance;

	scanf("%d %d", &money, &change);

	// 1차 Guard Clause: 애초에 출금하려는 금액이 가진 돈보다 많으면 즉시 차단
	if (money < change) {
		printf("잔액 부족\n");
		return 0;
	}

	// 출금 진행
	balance = money - change;

	// 수수료 부과 로직
	if (balance < 1000) {
		balance = balance - 500;
	}

	// 2차 Guard Clause: 수수료를 뗐더니 마이너스가 된 경우 즉시 차단
	if (balance < 0) {
		printf("잔액 부족\n");
		return 0;
	}

	// 모든 관문을 통과한 정상 결제
	printf("출금 성공, 잔액 : %d\n", balance);

	return 0;
}
```

## 5. 핵심 인사이트 (Takeaway)
아키텍처의 논리적 순서: 출력값이 맞더라도, 조건을 어떤 순서로 처리하는지가 시스템의 강건성에 영향을 준다. 비정상적인 데이터는 연산 깊숙이 들어오기 전에 가드 절(Guard Clause)로 조기 차단해야 한다.

변수의 상태 추적: 중간 결과값(balance)이 무엇을 의미하는지 분명히 하고, '1차 출금 직후 잔액'과 '수수료 차감 후 최종 잔액'의 상태 변화를 완벽하게 통제해야 한다.
