[실전 코딩 문제: 일본 EC 사이트 VIP 할인 및 배송비 산정 로직]
요구사항: 사용자가 장바구니에 담은 3개 상품의 가격을 입력받아, 할인과 배송비를 순차적으로 계산하여 최종 결제 금액을 출력하라.
입력: 3개 상품의 가격(엔화, 정수)을 공백으로 구분하여 입력받는다.
조건 1 (할인): 세 상품의 총금액이 10,000엔 이상이면, 총금액에서 10%를 할인한다. (단, 소수점 이하는 버리고 정수로 처리한다.)
조건 2 (배송비): (할인이 적용되었다면 적용된 후의) 금액이 15,000엔 미만이면, 배송비 500엔을 추가한다. 15,000엔 이상이면 배송비는 무료다.
출력: 할인 전 순수 '총 금액'과, 할인 및 배송비가 모두 적용된 '최종 결제 금액'을 각각 출력한다.

초기에 짠 코드
#define _CRT_SECURE_NO_WARNINGS#include <stdio.h>int main()
{
int goods1, goods2, goods3;
int total_amount;

printf("상품 가격 입력: ");
scanf("%d %d %d", &goods1, &goods2, &goods3);

total_amount = goods1 + goods2 + goods3;
printf("총 금액 : %d\n", total_amount);

if (total_amount >= 10000)
{
total_amount = total_amount * 0.9;

if (total_amount < 15000)
{
total_amount = total_amount + 500;
printf("최종 결제 금액 : %d", total_amount);

return 0;
}
else
{
printf("최종 결제 금액 : %d", total_amount);
}

}
else
{
total_amount = total_amount + 500;
printf("최종 결제 금액 : %d", total_amount);
}

return 0;

}
2. 초기 코드의 치명적 결함 (The Trap)
printf("최종 결제 금액 : %d", total_amount);라는 똑같은 코드가 무려 세번이나 반복한다. 만약 기획팀에서 "출력 문구를 '고객님의 최종 결제 금액은 X엔입니다'로 바꿔주세요"라고 할 경우 3번이나 수정 작업을 거쳐야 한다.
할인 받는 사람과 할인 받지 않는 사람을 쪼갠 뒤 각각 배송비를 계산하고 있다.
[현재 네 코드의 논리 흐름]
if (1만 엔 이상인가?)
예: 10% 할인 ➔ if (1.5만 엔 미만인가?) ➔ 배송비 추가 ➔ 출력
아니오: (1.5만 엔 이상인가?) ➔ 배송비 추가 안 함 ➔ 출력
else (1만 엔 미만인가?)
➔ 배송비 추가 ➔ 출력
이렇게 짤 경우 조건 하나가 더 추가 되면(예: 쿠폰 적용) 코드가 더욱 복잡해진다.

3. 단계별 디버깅 과정
printf("최종 결제 금액 : %d", total_amount);를 if문을 벗어나 결과가 나온 후 출력 되게 설정.
불필요한 else 나 if 문 제거.

4. 최종 최적화 코드 (Final Verified Code)
#define _CRT_SECURE_NO_WARNINGS
#include <stdio.h>

int main()
{
int goods1, goods2, goods3;
int total_amount;

printf("상품 가격 입력: ");
scanf("%d %d %d", &goods1, &goods2, &goods3);

total_amount = goods1 + goods2 + goods3;
printf("총 금액 : %d\n", total_amount);

if (total_amount >= 10000) //불필요하게 else 문을 사용하지 않아도 된다.
{
total_amount = (int)(total_amount * 0.9); //명시적으로 표시.
}
if (total_amount < 15000)
{
total_amount = total_amount + 500;
}

printf("최종 결제 금액 : %d", total_amount); //똑같은 문장을 규칙 마다 넣는거 보다 효율적으로 한개만 배치.

return 0;

}
5. 핵심 인사이트 (Takeaway)
DRY (Don't Repeat Yourself) 원칙 준수: printf와 같은 동일한 출력 로직이 반복되는 것을 방지하여 유지보수성을 높였다.
파이프라인(Pipeline) 패턴 적용: 불필요한 else 분기문을 제거하고, 데이터(total_amount)가 할인과 배송비라는 독립적인 공정을 순차적으로 거치며 가공되도록 제어 흐름을 최적화했다.

6.추가 참고 사항
total_amount = total_amount * 0.9; 경우 현재 요구 사항인 '소수점 이하 버림'이라는 것은 충족 시켰으나 암시적 형변환에 기댄 결과로 int 와 0.9가 연산되어 결과값이 나온뒤 다시 int를 통해서 소수점이 자려나간 결과다.
